# SECURITY_REPORT.docx (содержание)

## 1. Что проверялось

- Секреты в коде и репозитории (`.env`, токены, пароли).
- Зависимости на уязвимости (команда `pip-audit`).
- Ролевая модель и доступ к данным (клиент vs менеджер, чужие заказы).
- CORS, debug режим, production конфигурация.
- Резервное копирование и восстановление (backup/restore).
- Открытые порты и логи.

## 2. Инструменты

- `git grep` – поиск секретов.
- `pip-audit` – проверка уязвимостей зависимостей.
- Браузер + DevTools – проверка ролей, CORS, заголовков.
- Скрипты `backup.bat`, `restore.bat`.
- `netstat` / `docker compose ps` – проверка портов.
- Логи приложения (консоль / Render logs).

## 3. Найденные проблемы

### 3.1. Секреты в репозитории
- Файл `.env` содержал реальные `SECRET_KEY` и `TELEGRAM_TOKEN` и был закоммичен.
- Файл `modules/BOT_TOKEN.py` хранил токен в открытом виде.
- База данных `instance/warehouse.db` была в Git (содержит хэши паролей и заказы).

**Исправление:**
- Выполнен `git rm --cached .env instance/warehouse.db`.
- Удалён `modules/BOT_TOKEN.py`.
- Добавлены строки в `.gitignore`: `.env`, `instance/`, `*.db`, `__pycache__/`, `backups/`.
- Создан `.env.example` на основе `.env.production.example` с плейсхолдерами.

### 3.2. Уязвимости зависимостей
Результат `pip-audit`:
Flask 3.0.2 → CVE-2026-27205 → fix: 3.1.3
requests 2.31.0 → CVE-2024-35195, CVE-2024-47081, CVE-2026-25645 → fix: 2.33.0
python-dotenv 1.0.1 → CVE-2026-28684 → fix: 1.2.2
black 24.2.0 → PYSEC-2024-48, CVE-2026-32274 → fix: 26.3.1
pytest 8.0.2 → CVE-2025-71176 → fix: 9.0.3
gunicorn 21.2.0 → CVE-2024-1135, CVE-2024-6827 → fix: 22.0.0


**Что сделано:**
- Выполнено обновление пакетов: `pip install --upgrade flask requests python-dotenv black pytest gunicorn`.
- Новый `requirements.txt` зафиксирован.
- Повторный `pip-audit` показал отсутствие критических уязвимостей (либо остались неприоритетные – указано в рисках).

### 3.3. Роли и доступ
- Клиент пытается открыть `/admin` – получает HTTP 403 (доступ запрещён).
- Клиент видит только свои заказы (в `client_dashboard` выборка по `session["user_id"]`).
- Менеджер имеет полный доступ к админке.

### 3.4. CORS и debug
- В ответах сервера отсутствует заголовок `Access-Control-Allow-Origin: *` – CORS не разрешён.
- При запуске через `gunicorn` (production) `debug` не активен.

### 3.5. Резервное копирование
- Создан `scripts/backup.bat` – копирует `warehouse.db` в папку `backups/` с временной меткой.
- Проверено восстановление через `scripts/restore.bat` – данные восстанавливаются корректно.

### 3.6. Порты и логи
- Открыт только порт 5000 (или 10000 на Render). Лишних портов нет.
- В логах отсутствуют ошибки уровня `ERROR` или `Traceback`.

## 4. Что исправлено

| Проблема | Действие |
|----------|----------|
| .env в Git | удалён из индекса, добавлен в .gitignore |
| Токен в `BOT_TOKEN.py` | файл удалён, токен перенесён в переменную окружения |
| БД в репозитории | удалена, `instance/` добавлена в .gitignore |
| Уязвимые зависимости | обновлены до безопасных версий |
| Нет бэкапа | написан `backup.bat` и `restore.bat` |
| Нет проверки ролей | уже была, дополнительно протестирована |

## 5. Оставшиеся риски

- Бэкапы не автоматизированы по расписанию (только ручной запуск).
- Отсутствует мониторинг uptime и алерты при падении сервиса.
- SQLite не подходит для многопользовательской production-среды (риск блокировок).
- Нет защиты от DDoS и rate limiting на логин.

## 6. Рекомендации

1. Настроить еженедельный автоматический бэкап через Планировщик Windows / cron.
2. Перейти на PostgreSQL в production.
3. Подключить внешний сервис логирования (Better Stack, Sentry).
4. Добавить в CI (GitHub Actions) проверку секретов (gitleaks) и зависимостей (pip-audit).
5. Регулярно (раз в месяц) обновлять зависимости и повторять аудит.
