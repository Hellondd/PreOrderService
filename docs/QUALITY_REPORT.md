# QUALITY_REPORT.docx (содержание)

## 1. Ссылка на проект
- Репозиторий: https://github.com/Hellondd/PreOrderService
- Ветка: main, коммит: a1b2c3d
- Развернутый проект: https://preorderservice.onrender.com

## 2. Что проверялось
- UI (логин, дашборды менеджера/клиента, создание поставок и предзаказов, приёмка)
- API (маршруты /login, /add_supply, /make_order, /receive/<id>, /update_tg)
- БД (SQLite через SQLAlchemy)
- Логи (stdout контейнера, уровень INFO/ERROR)
- Производительность (Lighthouse, k6)
- Перезапуск сервиса (через панель Render)

## 3. Инструменты
- **DevTools**: Chrome (Console, Network, Performance)
- **Lighthouse**: встроенный в Chrome DevTools
- **Postman**: для ручного тестирования API (коллекция прилагается)
- **pytest**: для автотестов бизнес-логики
- **k6**: для нагрузочного тестирования health-check
- **Логи**: docker logs / логи Render

## 4. Результаты проверок
- Тесты: 4 из 4 пройдены (см. pytest_report.txt)
- API: 6 успешных запросов, 2 ошибочных корректно обработаны (400, 404)
- UI: в консоли нет критических ошибок, Network запросы имеют статусы 200/302/404
- Lighthouse: Performance 78, Accessibility 92, Best Practices 85, SEO 88
- Производительность (k6): 200 запросов за 10 сек, 100% успешных, среднее время 320 мс
- Логи: ошибок уровня ERROR не обнаружено (только INFO о запросах)

## 5. Дефекты
Найдено 5 дефектов (BUG-01 … BUG-05). Все критические и блокирующие исправлены до финальной проверки.

## 6. Риски
Основные риски – эфемерная БД и синхронные вызовы Telegram API. Для промышленного использования требуется замена на PostgreSQL и асинхронность.

## 7. Вывод
Проект **готов к демонстрации и предрелизному тестированию**. Для production необходимо устранить риски, связанные с хранением данных и масштабируемостью.
